Bybit v5 Open API allow users to implement trade by one collection API.

## Scope
| Account Mode | v5 Open API | Comments|
|:----- |:-------|------ |
|Normal account |true |can trade spot,futures,option |
|Unified margin account |true |can trade linear futures,option; cannot trade spot,inverse|
|Unified trade account |true |can trade everything|

### Parameters for Authenticated Endpoints
The following parameters must be used for authentication:

* `api_key`
* `timestamp` - UTC timestamp in milliseconds
* `sign` - a signature derived from the request's parameters<br/>

We also provide `recv_window` (unit in millisecond and default value is 5,000) to specify how long an HTTP request is valid. It is also used to prevent replay attacks.

A smaller `recv_window` is more secure, but your request may fail if the transmission time is greater than your `recv_window`.

Please make sure that your timestamp is in sync with our server time. You can use the [Server Time endpoint](../v3/spot/rest-public/server-time#) to check your own machine time.

:::caution
Please make sure that the timestamp parameter adheres to the **following rule**:<br/>
_server_time - recv_window <= timestamp < server_time + 1000_<br/>
server_time stands for Bybit server time, which can be queried via the [Server Time endpoint](../v3/spot/rest-public/server-time#).
:::

### Create A Request

:::tip
To assist in diagnosing advanced network problems, you may consider adding `cdn-request-id` to your request headers. Its value should be unique for each request.
:::

:::info
Example signature algorithms can be found <a href="https://github.com/bybit-exchange/api-usage-examples">here</a>
:::

**Basic steps:**
1. timestamp + api-key + (recv_window) + (queryString | jsonBodyString)<br/>
2. Use the HMAC_SHA256 algorithm to sign the string in step 1, and convert it to a hex string to obtain the sign parameter.<br/>
3. Append the `sign` parameter to request header, and send the HTTP request.
**Note:** the plain text for **GET** and **POST** requests is different. Please refer to blew examples.

> An example for how to generate plain text to encrypt

<Tabs>
<TabItem value="GET" label="GET">

```shell
# rule:
timestamp+api_key+recv_window+queryString

# param_str
"1658384314791XXXXXXXXXX5000category=option&symbol=BTC-29JUL22-25000-C"

# parse
timestamp = "1658384314791"
api_key = "XXXXXXXXXX"
recv_window = "5000"
queryString = "category=option&symbol=BTC-29JUL22-25000-C"
```

</TabItem>

<TabItem value="POST" label="POST">

```shell
# rule:
timestamp+api_key+recv_window+raw_request_body

# param_str
1658385579423XXXXXXXXXX5000{
"category": "option"
}

# parse
timestamp = 1658385579423
api_key = XXXXXXXXXX
recv_window = 5000
raw_request_body = {"category": "option"}
```

</TabItem>
</Tabs>

## Common response parameters
| Parameter | Type | Comments|
|:----- |:-----|----- |
|retCode |number |Success/Error code |
|retMsg |string |Success/Error msg. Can be `OK`,`success`,`Success` for Success message|
|result |Object |Business data result|
|retExtInfo |Object |Extend info. Most of time, it is `{}`|
|time |number |Current timestamp (ms)|

```json
{
    "retCode": 0,
    "retMsg": "OK",
    "result": {
    },
    "retExtInfo": {},
    "time": 1671017382656
}
```
