Query unfilled or partially filled orders in **real-time**. To query longer order records, please use the [order history](order-list#) interface.
> **Cover: Spot (Unified Account) / Linear Perpetual (Unified Account) / Option (Unified Account)**

:::tip
- It also supports query filled, cancelled and rejected orders, which occurred in last 10 minutes. At most 500 orders for each UID will be returned.
- You can query by symbol, baseCoin, orderId and orderLinkId, and if you pass multiple params, system will process them according to this priority: orderId > orderLinkId > symbol > baseCoin.
- The records are sort by the `createdTime` from newest to oldest.
:::

### HTTP Request
GET `/v5/order/realtime`

### Request Parameters
| Parameter | Required | Type | Comments|
|:----- |:-------|:-----|------ |
|category |**true** |string |Product type. `spot`,`linear`,`option` |
|symbol |false |string |Symbol name |
|baseCoin |false |string |Base coin. for `option` ONLY. Return all option open orders if not passed|
|settleCoin |false |string |Settle coin. for `linear` ONLY|
|orderId |false |string |Order id |
|orderLinkId |false |string |User customised order id |
|openOnly |false |integer |<li>`0`(default): query open orders ONLY</li><li>`1`: return cancelled, rejected or totally filled orders by last 10 minutes, A maximum of 500 records are kept under each account. If the Bybit service is restarted due to an update, this part of the data will be cleared and accumulated again, but the order records will still be queried in [order history](order-list#)</li>|
|orderFilter |false |string|`Order`: active order, `StopOrder`: conditional order, `tpslOrder`: spot tp/sl order. default: all kinds of orders|
|limit |false |integere |Limit for data size per page. [1,50]. default: `20`|
|cursor |false |string |Cursor. Used to pagination|
